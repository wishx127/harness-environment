# prompt.md

## 修改范围原则

默认策略：
  - 优先最小可行修改（minimal change）
  - 避免无关重构
  - 避免改变未被任务涉及的模块
  - 避免引入新抽象层，除非必要

## 硬性约束
- **性能**：
  - 避免过早优化
  - 优先保证正确性与可读性
  - 仅在性能成为瓶颈时进行结构性优化

  前端：
    - 避免阻塞渲染
    - 避免不必要的大型依赖
    - 控制 bundle 体积增长
    - 优先增量修改而非重写

  后端：
    - 避免明显的性能反模式
    - 避免不必要的同步阻塞
    - 优先 O(n) 或更优复杂度
- **风格**：统一代码风格与格式，避免风格一致性问题
- **安全**：
  禁止：
    - 执行破坏性操作
    - 修改关键配置文件（除非明确要求）
    - 修改数据库 schema（除非明确要求）
- **可维护**：结构清晰，支持后续扩展与迭代

## 优先级
当约束冲突时，按以下顺序决策：

1. 正确性（correctness）
2. 可验证性（verifiability）
3. 安全性（safety）
4. 可维护性（maintainability）
5. 性能（performance）
6. 体验简洁性（clarity）
7. 风格一致性（style consistency）

## 非目标
- 不涵盖具体实现细节与技术方案
- 不替代需求文档、设计文档或测试计划
- 不定义团队外部流程

## 不确定时

当信息不足时：
  - 不假设不存在的需求
  - 引入人工干预
